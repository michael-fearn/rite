# Migrate the Service-layer Acceptance Test to the Acceptance Test Lifecycle

## What to build

Migrate the Service-layer Acceptance Test onto the existing Acceptance Test Lifecycle module proven by the NFS shared-mount tracer bullet, while preserving its operator-facing behavior and keeping Service-layer-specific artifacts and proof semantics in the Service-layer workflow.

The lifecycle Module should own the common Acceptance Test path for the Service-layer workflow: resolving Host, Template, NAS Endpoint, Acceptance Policy, Primary Acceptance VM, Peer Acceptance VM, Ephemeral Dataset, and Mount intent; writing common generated Dataset and VM artifacts; refusing to overwrite common generated artifacts; running NAS Reconcile create/destroy and cleanup postcondition assertions; provisioning generated Acceptance VMs; running retrying SSH checks; and performing default shared-resource cleanup.

The Service-layer workflow should continue to own its temporary Service artifacts, Service Sibling SOPS File creation/encryption, Quadlet Fragment, Native Service templates, Service Deploy calls, TrueNAS NFS reload proof, and contract-specific checks for Service Secrets, Container Aliases, Published Ports, Native Service behavior, Share-backed Volumes, and post-reload data flow.

- [ ] Overwrite refusal covers both lifecycle-owned artifacts and Service-layer-owned generated Service artifacts before any mutation.
- [ ] NAS Reconcile create, Derived Share assertion, NAS Reconcile destroy, and cleanup postcondition assertions use the lifecycle Module.
- [ ] Generated Acceptance VM provisioning uses lifecycle-owned behavior or an equivalent lifecycle helper while preserving current `auto_confirm` behavior and progress output expected by tests.
