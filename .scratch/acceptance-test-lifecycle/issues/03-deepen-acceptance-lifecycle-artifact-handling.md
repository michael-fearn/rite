# Deepen Acceptance Test Lifecycle artifact handling

## What to build

Move generated Acceptance artifact discovery, classification, refusal, cleanup, and failure-preservation mechanics into the importable `fortress_acceptance.lifecycle` Module.

The operator-facing scripts should stay thin. `scripts/acceptance-clean-generated-artifacts` should become a command-line entry point over lifecycle-owned artifact planning and deletion. Individual Acceptance Tests should be able to register workflow-specific generated artifacts with the lifecycle Module so the lifecycle owns common refusal, cleanup, and preservation mechanics without owning what those artifacts mean.

This is a deepening refactor: the Acceptance Test Lifecycle Interface should provide more leverage and locality for generated artifacts while keeping workflow-specific proof semantics owned by the individual Acceptance Tests.

## Current friction

The generated artifact set for Service-layer Acceptance is currently known in multiple places:

- `scripts/acceptance-service-layer` refuses and deletes Service-layer-specific generated artifacts.
- `scripts/acceptance-clean-generated-artifacts` independently discovers and classifies Service-layer artifacts for manual cleanup.
- `fortress_acceptance.lifecycle` owns common generated Ephemeral Dataset and Operational VM cleanup, but not the reusable artifact discovery/classification mechanics.

That split makes future Acceptance Tests harder to add safely: every new workflow-specific generated artifact risks needing parallel updates in workflow cleanup, failure-preservation behavior, and the operator cleanup command.

## Desired shape

The Acceptance Test Lifecycle Module should own generic generated artifact mechanics:

- representing expected generated artifacts
- deriving common VM and Ephemeral Dataset artifacts from Acceptance Policy data
- accepting workflow-specific artifact registrations from individual Acceptance Tests
- planning artifact cleanup as delete / absent / refuse
- refusing ordinary or ambiguous Inventory
- deleting planned generated repository artifacts
- providing shared helpers for preflight overwrite refusal and failure cleanup

Individual Acceptance Tests still own workflow-specific content and proof:

- Service-layer writes temporary Service YAML, Service Sibling SOPS File, Quadlet Fragment directory, Native Service YAML, and Native template directory.
- Service-layer encrypts the Service Sibling SOPS File through the ordinary SOPS ceremony.
- Service-layer runs Service Deploy, TrueNAS NFS service reload proof, and Service-specific checks.
- NFS shared-mount owns its read/write/delete proof semantics.

- [x] The lifecycle Module can derive NFS shared-mount generated VM artifacts and generated Ephemeral Dataset artifacts from the Acceptance Policy.

- [x] Service-layer failure cleanup uses lifecycle-owned artifact mechanics rather than maintaining a separate Service-layer artifact deletion path.
- [x] `keep_on_fail=true` still preserves generated Service artifacts, generated VM inventory/SOPS artifacts, provisioned VMs, and NAS acceptance resources for inspection.
- [x] NFS shared-mount workflow behavior remains compatible with current operator-facing CLI, prompt, progress output, cleanup behavior, and tests.
- [x] Service-layer workflow behavior remains compatible with current operator-facing CLI, prompt, progress output, command ordering, TrueNAS reload behavior, cleanup behavior, and tests.
- [x] Automated tests cover lifecycle-level artifact planning for NFS shared-mount and Service-layer workflows.
- [x] Automated tests cover the thin cleanup script entry point, including confirmation behavior, auto-confirm behavior, absent artifacts, deletion, and refusal of ordinary Inventory.
- [x] No workflow-specific proof semantics move into the lifecycle Module.

## Non-goals

- Do not make the Acceptance Test Lifecycle own temporary Service YAML/template content.
- Do not move Service-layer Service Deploy, TrueNAS reload proof, Service Secret checks, Container Alias checks, Published Port checks, Native Service checks, or post-reload checks into the lifecycle Module.
- Do not redesign NAS Reconcile, Service Deploy, or Host Readiness.
- Do not expand generated artifact cleanup to destroy live Proxmox VMs, TrueNAS Datasets, or NFS Shares.
