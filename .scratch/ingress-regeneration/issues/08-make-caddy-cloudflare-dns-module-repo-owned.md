## What to build

Make the Caddy binary used by the Ingress a fortress-owned artifact that durably includes the Cloudflare DNS provider module.

During the live Ingress proof on 2026-05-14, `service-deploy internal-ingress` successfully rendered the Cloudflare token into Caddy's root-owned environment file, but Caddy could not adapt generated routes containing:

```caddy
tls {
	dns cloudflare {$CLOUDFLARE_API_TOKEN}
}
```

The installed package binary did not include `dns.providers.cloudflare`. The live VM was repaired manually with:

```sh
sudo caddy add-package github.com/caddy-dns/cloudflare
```

That proves the dependency, but it is not durable. A future package reinstall, package upgrade, VM rebuild, or Service Deploy behavior change could replace `/usr/bin/caddy` with a stock binary and silently remove the Cloudflare module.

## Agent Brief

**Category:** enhancement

**Summary:** Own the Caddy build/install path so the Ingress Caddy binary always includes `dns.providers.cloudflare`.

**Current behavior:**

The Native Service declaration for the Ingress installs Caddy from the configured apt repository. That package provides a stock Caddy binary. Generated Caddy routes require the Cloudflare DNS provider for Let's Encrypt DNS-01, but the stock binary does not include `dns.providers.cloudflare`.

The live VM currently works only because the binary was manually replaced with `caddy add-package github.com/caddy-dns/cloudflare`. Fortress inventory and Service Deploy do not declare or converge that module.

**Desired behavior:**

Fortress should declare and converge the Caddy Cloudflare DNS provider as part of the Ingress Native Service lifecycle. After any fresh VM build, package reinstall, package upgrade, or `service-deploy internal-ingress`, the deployed Caddy binary should include `dns.providers.cloudflare` before Caddy validates or starts generated routes that use `dns cloudflare`.

The convergence path should be idempotent and testable without printing secret values. If module installation or verification fails, Service Deploy should fail before leaving Caddy in a misleading "successfully deployed" state.

**Key interfaces:**

- Native Service inventory shape — may need a way to declare required Caddy modules or post-install package extensions for Native Services.
- Service Deploy workflow — should install/build/verify required Native Service package extensions before rendering/restarting dependent config.
- Ingress Native Service declaration — should declare the Cloudflare DNS provider requirement for Caddy.
- Caddy validation/checking — should verify that `dns.providers.cloudflare` is available without requiring the Cloudflare token value to be printed.

**Acceptance criteria:**

- [ ] `internal-ingress` declares, directly or through an established Native Service extension mechanism, that Caddy requires `github.com/caddy-dns/cloudflare` / `dns.providers.cloudflare`.
- [ ] `service-deploy internal-ingress` converges the required Caddy Cloudflare DNS provider on a fresh Ingress Backend VM without manual `caddy add-package` intervention.
- [ ] The deploy path is idempotent when the module is already present.
- [ ] The deploy path verifies `dns.providers.cloudflare` is present before restarting or reporting success for Caddy.
- [ ] A stock Caddy binary without `dns.providers.cloudflare` is caught by automated tests or a workflow fixture.
- [ ] Tests cover the generated Ansible/workflow behavior without requiring real Cloudflare credentials or printing `CLOUDFLARE_API_TOKEN`.
- [ ] Documentation or runbook notes explain that the Cloudflare DNS module is repo-owned and should not be repaired manually with `caddy add-package` except as an emergency break-glass step.

**Out of scope:**

- Rotating or changing the Cloudflare API Token.
- Changing the generated Caddy route syntax for DNS-01.
- Replacing Native Service Environment Secrets.
- Proving live Let's Encrypt issuance again; this issue is about making the Caddy module installation durable.

## Notes From Live Proof

- `caddy list-modules` initially did not show `dns.providers.cloudflare`.
- `caddy validate --config /etc/caddy/Caddyfile` failed with `module not registered: dns.providers.cloudflare`.
- After manual `sudo caddy add-package github.com/caddy-dns/cloudflare`, module verification succeeded.
- A separate systemd EnvironmentFile drop-in now loads `/etc/default/caddy` and overrides the stock unit's `--environ` flag to avoid future environment logging.
