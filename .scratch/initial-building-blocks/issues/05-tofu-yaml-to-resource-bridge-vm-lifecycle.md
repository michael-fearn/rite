## What to build

OpenTofu reads the same VM yaml files Ansible uses; provider aliases come from the host yaml directory; PVE tokens decrypted only into ephemeral env vars by a wrapper. The full VM lifecycle command sequences prepare → tofu apply → configure into one operator command. Destroy refuses while services still reference the VM.

- OpenTofu provider aliases are not dynamically indexable as sketched in older architecture notes; use a generated/static provider mapping pattern rather than pretending pure HCL can create provider blocks from YAML.

Provider-alias patterns considered:

- **Generated provider config from inventory:** a wrapper or generator reads `inventory/hosts/*.yaml` and writes a small ignored or committed `.tf` file containing one provider alias and token variable binding per Host. This preserves one state file and lets YAML remain the source of truth, but adds generated HCL that must be refreshed before `tofu init/plan/apply`.
- **Static root module provider aliases:** hand-maintain provider blocks for known Hosts in the root tofu module. This is simple and explicit for a four-Host fleet, but adding a Host requires a YAML change plus a small HCL change.
- **One tofu root per Host:** avoid aliases by running tofu separately per Host. This avoids the provider-alias limitation, but violates the desired single fleet state and makes cross-Host planning harder.
- **External orchestrator:** generate full tofu working directories or run per-Host plans from Python/Bash. This is flexible, but it moves too much lifecycle logic outside tofu for this slice.

Recommended pattern for this issue: generate or maintain static root-module provider aliases from Host YAML, then partition VM resources by `placement.host` so each resource block can bind to a literal provider alias. Keep one fleet state file. The implementation should document whichever static/generated pattern it chooses and include tests that catch missing Host alias coverage.

## Agent Brief

**Category:** enhancement
**Summary:** Add the OpenTofu VM resource bridge and VM lifecycle commands.

**Current behavior:**
The repository has YAML inventory schemas, cross-file validation, Host workflows, PVE token creation, and Template building. The `tofu` module is effectively empty, and `vm-up` / `vm-destroy` are still TODO operator commands. VM YAML files already express placement, source Template, hardware, network, cloud-init hostname, optional NFS Mounts, backup intent, and a plaintext `ssh_public_key` field populated by Prepare.

**Desired behavior:**
OpenTofu provisions VM shells from the same VM YAML files consumed by Ansible. The VM lifecycle is exposed through intent-named `just` commands:

- `just vm-up vm=<name>` runs Prepare, shows an explicit tofu plan/approval flow, applies the selected VM, then configures the VM through Ansible.
- `just vm-destroy vm=<name>` refuses if any Service still references the VM, then destroys the Proxmox VM through tofu. It must leave the VM YAML and VM Sibling SOPS File in the repo.

Prepare generates the per-VM SSH keypair, writes the private key encrypted to the VM Sibling SOPS File, writes the public key plaintext into VM YAML, and refuses if the VM Sibling SOPS File already exists. Configure waits for cloud-init/SSH readiness before finalizing the admin user and any first-boot VM configuration owned by this slice.

Tofu must never read SOPS. A wrapper decrypts only current PVE API tokens from Host Sibling SOPS Files into ephemeral environment variables, invokes tofu, and cleans up. Token variables must be marked sensitive, and secrets must not appear in plan output or state.

**Key interfaces:**
- VM YAML under `inventory/vms/<name>.yaml` is the VM source of truth.
- Host YAML under `inventory/hosts/<name>.yaml` provides PVE endpoint and provider-alias input.
- Host Sibling SOPS Files provide the current tofu PVE token produced by Host Configure.
- VM Sibling SOPS Files store VM SSH private key material.
- OpenTofu uses the `bpg/proxmox` provider and one fleet state file.
- Provider aliases must be static/generated HCL, not dynamically created at runtime inside pure HCL. The implementation should either generate provider alias config from Host YAML or maintain explicit aliases for the declared fleet, and should test that every `placement.host` has provider coverage.
- Cloud-init userdata is assembled from VM YAML plus the plaintext `ssh_public_key`.

**Acceptance criteria:**
- [ ] The tofu root/module reads VM YAML files and creates Proxmox VM resources for declared VMs.
- [ ] VM resources are associated with the correct Host provider alias based on `placement.host`.
- [ ] The provider alias strategy is documented and tested for missing Host alias coverage.
- [ ] One fleet state file is used for all Hosts.
- [ ] The tofu wrapper decrypts PVE API tokens only into ephemeral environment variables and invokes tofu as the supported entry point.
- [ ] PVE token variables are declared `sensitive = true`.
- [ ] Tests or fixture plans verify tokens do not appear in plan output or tfstate.
- [ ] Cloud-init userdata includes the VM hostname, admin user bootstrap material, network intent needed at first boot, and the VM `ssh_public_key`.
- [ ] Prepare generates a per-VM SSH keypair, writes the private key encrypted to the VM Sibling SOPS File, and writes the public key plaintext into VM YAML.
- [ ] Prepare refuses if the VM Sibling SOPS File already exists.
- [ ] Configure waits for cloud-init/SSH readiness before applying VM configuration tasks.
- [ ] `just vm-up vm=<name>` runs Prepare, tofu plan with explicit approval, tofu apply, and Configure.
- [ ] `just vm-destroy vm=<name>` refuses when any Service references the VM as a Backend.
- [ ] `just vm-destroy vm=<name>` destroys the Proxmox VM through tofu and leaves the VM YAML and VM Sibling SOPS File untouched.
- [ ] The wrapper reads the current PVE token shape produced by Host Configure; token rotation remains out of scope.
- [ ] `runbooks/new-vm.md` documents adding a VM, Prepare behavior, approval flow, Configure behavior, destroy behavior, and when to alert the maintainer.
- [ ] End-to-end testing can be driven through `just`; if real Host access or manual approval is required, the agent reports the exact command/output and waits for the maintainer.

**Out of scope:**
- PVE token rotation or version migration.
- Service deployment, ingress regeneration, and Service data migration.
- Deleting VM YAML or VM Sibling SOPS Files during destroy.
- Multi-VM Service Backend support beyond refusing singular Service references that already exist.
- Building or rebuilding Templates.
- Managing Host storage creation.
