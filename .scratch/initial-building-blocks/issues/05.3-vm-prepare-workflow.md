## What to build

Add VM Prepare as a refusal-style local workflow. Prepare generates the per-VM admin SSH keypair, writes the private key encrypted to the VM Sibling SOPS File, and writes the public key plaintext into the VM YAML for future cloud-init use.

## Acceptance criteria

- [ ] `just vm-prepare vm=<name>` or a script invoked by future `just vm-up` exists for VM Prepare.
- [ ] The Prepare workflow rejects undeclared VMs.
- [ ] Prepare refuses if `inventory/vms/<vm>.sops.yaml` already exists.
- [ ] Prepare generates an ed25519 SSH keypair for the VM admin user.
- [ ] Prepare writes the private key into `inventory/vms/<vm>.sops.yaml` encrypted via SOPS.
- [ ] Prepare writes the public key plaintext into the declared VM YAML `cloud_init.ssh_public_key` field or the existing schema field used for that purpose.
- [ ] Prepare preserves unrelated VM YAML fields and comments as far as the repo's YAML tooling reasonably allows.
- [ ] Prepare deletes plaintext private-key tempfiles before exit.
- [ ] Error messages point the operator at the future VM admin-key rotation workflow instead of overwriting existing key material.
- [ ] Tests use fake `ssh-keygen` and fake `sops` to cover success, refusal when the Sibling SOPS File exists, undeclared VM rejection, and YAML public-key update.
- [ ] Tests verify no Proxmox, tofu, or SSH-to-VM access is required.

## Blocked by

None

## Design decisions

- Prepare and Configure are separate because Prepare creates key material and refuses to re-run, while Configure is idempotent.
- The VM Sibling SOPS File is sparse and exists only because this workflow creates VM SSH private key material.
- The plaintext public key in VM YAML is intentional because cloud-init needs it and it is not secret.

## Out of scope

- OpenTofu plan/apply.
- Waiting for cloud-init.
- Final VM admin user configuration after first boot.
- VM admin key rotation.
