## What to build

Add the first real OpenTofu VM resource bridge for a selected VM. Tofu reads VM YAML, uses generated static Host partitions/provider aliases, assembles cloud-init userdata from VM YAML plus the plaintext VM public key, and provisions only the selected VM while preserving one root module and one fleet state file.

## Acceptance criteria

- [ ] The tofu module creates Proxmox VM resources from declared VM YAML.
- [ ] A selected VM variable or equivalent selection mechanism ensures early `vm-up` plans affect only the requested VM.
- [ ] Selection preserves one root module and one fleet state file; it must not create one tofu root per VM or Host.
- [ ] VM resources bind to the correct literal Host provider alias through the generated per-Host partition pattern.
- [ ] The implementation avoids dynamic provider indexing.
- [ ] Cloud-init userdata includes hostname, admin bootstrap user material, network intent needed at first boot, and the VM plaintext SSH public key.
- [ ] Missing VM public key fails clearly and instructs the operator to run Prepare first.
- [ ] Hardware, disk, network, Template source, and placement fields are read from VM YAML according to the current schema.
- [ ] Tofu variables for PVE tokens remain sensitive.
- [ ] Tests or fixture plans verify the selected VM filter does not plan unrelated VMs.
- [ ] Tests or fixture plans verify token values do not appear in plan output or tfstate fixtures.
- [ ] Documentation explains that `just` is the operator surface and the tofu wrapper is the implementation entry point.

## Design decisions

- The first destructive VM resource slice targets exactly one selected VM with care.
- The long-term shape remains one fleet state file.
- `just` will wrap scripts for the unified operator interface; scripts remain safe direct debugging entry points.

## Out of scope

- End-to-end `vm-up` orchestration.
- VM Configure.
- Destroy workflow.
- Service deployment or ingress changes.
- Template building.
