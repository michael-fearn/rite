## What to build

Add the idempotent VM Configure workflow. Configure waits for a provisioned VM to finish cloud-init and become reachable over SSH, then finalizes the VM admin user and first-boot VM configuration owned by this slice.

## Acceptance criteria

- [x] `just vm-configure vm=<name>` or a script invoked by future `just vm-up` exists for VM Configure.
- [x] Configure rejects undeclared VMs.
- [x] Configure requires the VM Sibling SOPS File created by Prepare.
- [x] Configure uses the existing tmpfs SSH key decryption pattern rather than writing private keys to persistent disk.
- [x] Configure waits for SSH readiness.
- [x] Configure waits for cloud-init completion before applying post-boot tasks.
- [x] Configure finalizes the admin user according to the VM lifecycle design.
- [x] Configure is re-runnable after the first successful run.
- [x] Tests cover command wiring, missing Sibling SOPS File failure, cloud-init wait task presence, and tmpfs key wrapper usage.
- [x] Ansible syntax or equivalent workflow validation passes for new playbooks/roles.

## Blocked by

.scratch/initial-building-blocks/issues/05.3-vm-prepare-workflow.md

## Design decisions

- Configure is separate from Prepare because Configure is idempotent and talks to a booted VM.
- Cloud-init owns only first-boot bootstrap material; Ansible owns the rest.

## Out of scope

- OpenTofu plan/apply.
- Service deployment.
- NFS Mount implementation beyond what is explicitly owned by this VM lifecycle slice.
- VM admin key rotation.
