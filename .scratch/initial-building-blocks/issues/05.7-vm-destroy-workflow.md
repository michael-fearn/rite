## What to build

Replace the `vm-destroy` stub with a guarded destroy workflow. `just vm-destroy vm=<name>` refuses if any Service still references the VM as a Backend, then destroys only the provisioned Proxmox VM through tofu. It must leave the VM YAML and VM Sibling SOPS File in the repo.

## Acceptance criteria

- [ ] `just vm-destroy vm=<name>` invokes a real script such as `scripts/vm-destroy <name>`.
- [ ] The script rejects undeclared VMs.
- [ ] The script scans `inventory/services/*.yaml` for singular `backend.vm` references to the selected VM.
- [ ] Destroy refuses clearly when any Service Backend references the VM.
- [ ] Destroy runs selected-VM tofu destroy through the tofu wrapper only after preflight passes.
- [ ] Destroy affects only the selected VM resource.
- [ ] Destroy leaves `inventory/vms/<vm>.yaml` untouched.
- [ ] Destroy leaves `inventory/vms/<vm>.sops.yaml` untouched.
- [ ] Tests cover referenced-VM refusal, unreferenced destroy command wiring, undeclared VM rejection, and preservation of VM YAML/Sibling SOPS File.
- [ ] Documentation states that deleting repo files is a separate human cleanup decision, not part of lifecycle destroy.

## Design decisions

- Destroy removes the provisioned Proxmox VM only.
- VM declarations and VM Sibling SOPS Files stay in git after destroy.
- Service Backend protection is required before destructive tofu operations.

## Out of scope

- Deleting VM YAML.
- Deleting VM Sibling SOPS Files.
- Service migration or Service deletion.
- Multi-VM Backend support beyond refusing singular Service references that already exist.
