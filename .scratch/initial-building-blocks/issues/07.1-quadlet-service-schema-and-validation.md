## What to build

Define the issue-07 Quadlet Service yaml contract and validation rules. The schema should replace the scaffolded pre-issue-07 shape rather than supporting a compatibility layer, and the validator should catch cross-file failures before any deploy reaches a VM.

## Acceptance criteria

- [ ] `deploy.type: quadlet` schema supports hostname, optional `service_group`, singular Backend, Ingress block, optional Service Data Owner, and containers with image, structured Published Ports, source/target volumes, non-secret env, Service Secret references, and Container Dependencies
- [ ] Ingress defaults are implemented: `enabled: true` when `hostname` is present, `enabled: false` when absent; enabled Ingress defaults to `exposure: lan_only`, `tls: letsencrypt_dns`, and `auth.type: none`
- [ ] Validator requires `hostname` only for ingress-enabled Services and checks duplicate hostnames only across ingress-enabled Services
- [ ] Validator rejects missing Backend VMs and treats Backend as singular for issue 07
- [ ] Published Port validation supports TCP, UDP, and TCP/UDP, defaults bind to `127.0.0.1`, requires explicit `ingress: true` for the HTTP-family Ingress Backend, and rejects all Published Port collisions on the same VM
- [ ] Image validation allows either a non-`latest` tag or digest and rejects untagged images and `:latest`
- [ ] Service Group validation treats `service_group` as a globally unique Service Group name, rejects groups spanning multiple Backend VMs, and rejects Container Alias collisions within each isolated or Service Group network
- [ ] Container Dependency validation allows only same-Service container references and rejects dependency cycles without promising application readiness
- [ ] Existing service fixtures are rewritten to the new schema; no compatibility layer for the scaffolded `ports`, `host` volume, `env_from_secrets`, or `requires_mounts` shape is added
