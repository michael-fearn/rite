## What to build

Render validated Quadlet Services into rootful system Quadlet units with deterministic fortress-owned runtime names, Service or Service Group networks, Service-owned volumes, Export-backed Volume dependencies, and same-Service container ordering.

## Acceptance criteria

- [ ] Quadlet renderer produces rootful system `.container` and `.network` files from the new issue-07 Service schema
- [ ] Runtime artifacts use `fortress-` names: container units and Podman container names as `fortress-<service>-<container>`, isolated networks as `fortress-<service>`, and Service Group networks as `fortress-group-<service_group>`
- [ ] Container `name` renders as the Podman network DNS alias while rendered container identity remains service-scoped
- [ ] Services get an isolated Podman network by default; Services in the same Service Group render against the shared group network
- [ ] Service-owned volumes render from `source.service_path` to `/srv/services/<service>/<service_path>` and use `target` as the container path
- [ ] Optional Service Data Owner (`storage.uid`/`storage.gid`) creates/chowns Service-owned paths only; Export-backed Volume ownership remains governed by VM Mount/NAS conventions
- [ ] Export-backed Volumes render from the Backend VM's declared Mount for `source.export`; omitting the source subpath binds the root of the Mount
- [ ] Export-backed Volumes add `Requires=` and `After=` dependencies on the matching `.mount` unit
- [ ] `depends_on` renders start-order/stop-coupling systemd unit dependencies between same-Service containers without modeling readiness
- [ ] Auto-update is disabled and no Podman auto-update behavior is emitted
- [ ] Golden-file tests cover single-container, multi-container, isolated network, Service Group network, Service-owned volume, Export-backed root Mount volume, Service Data Owner, dependency ordering, and image pinning variants
