## What to build

Add validated native Quadlet Fragment sidecar files for options fortress does not model directly, without allowing fragments to override fortress-owned invariants generated by the renderer.

## Acceptance criteria

- [ ] Quadlet Fragments live under `inventory/services/<service>.quadlet.d/`
- [ ] Container fragments are named `<container>.container`; optional network fragment is named `network.network`
- [ ] Unknown fragment filenames are invalid, including stale fragments for containers no longer declared by the Service yaml
- [ ] Fragments use native Quadlet/systemd INI syntax and are merged into the generated final Quadlet files
- [ ] Fragment validation derives forbidden keys from the generated Quadlet for that Service/container so generated scalar ownership cannot be overridden
- [ ] Repeated additive keys such as `Unit.Requires`, `Unit.After`, and `Unit.Wants` may add values without replacing or removing fortress-generated dependencies
- [ ] Fragments cannot override fortress-owned image, container identity, network, published ports, volumes, secrets, install behavior, auto-update policy, or generated dependencies
- [ ] Fragments are plaintext-only and must not contain Service secret values; all Service secrets go through the Service Sibling SOPS File and `_FILE` injection
- [ ] Golden-file tests prove fragment merging, unknown filename rejection, forbidden key rejection, and additive dependency merging
