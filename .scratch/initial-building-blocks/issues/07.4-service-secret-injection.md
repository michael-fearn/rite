## Agent Brief

**Category:** enhancement
**Summary:** Add Service Secret installation and `_FILE` environment injection for Quadlet Services.

**Current behavior:**
Fortress has the issue-07 Quadlet Service shape in place, including container-level `env` and `secrets` declarations. Quadlet rendering currently handles runtime names, networks, published ports, volumes, dependencies, and fragments, but it does not yet render non-secret environment variables, Podman secret bindings, or generated `_FILE` environment variables. The Service deployment workflow requires the Backend VM Sibling SOPS File for SSH, but it does not yet require or decrypt the Service Sibling SOPS File when a Service declares Service Secrets.

**Desired behavior:**
Service Secrets should live in the Service's Sibling SOPS File under top-level `secrets:`. A Quadlet container may declare `secrets` entries that reference those values by name and declare the `_FILE` environment variable the application will read.

Each declared Service Secret should be installed on the Backend VM as a Podman secret with a service-scoped name such as `fortress_<service>_<secret>`. Rendered containers should receive a Podman `Secret=` binding for the service-scoped secret and an environment variable whose name ends in `_FILE` and whose value points at `/run/secrets/<service-scoped-secret-name>`.

Secret values must never appear in rendered Quadlet artifacts, plaintext Service yaml, command output, or ordinary environment variables. Non-secret `env` remains declared in the Service yaml and should render as normal container environment.

`service-deploy` should require the Service Sibling SOPS File only when the selected Service declares Service Secrets. Services with no declared Service Secrets should not require a Service Sibling SOPS File.

Validation should reject ambiguous or unsafe environment ownership. Service yaml `env`, generated secret `_FILE` environment variables, and Quadlet Fragment `Environment=` entries must not override each other. Container secret environment variables must end in `_FILE`.

**Key interfaces:**
- Quadlet Service schema/container config shape — container `secrets` entries reference `secrets.<name>` from the Service Sibling SOPS File and declare the generated `_FILE` environment variable name.
- Service Sibling SOPS File shape — top-level `secrets:` map containing Service Secret values by name.
- Quadlet rendering API — renders non-secret `env`, generated `_FILE` environment variables, and Podman secret bindings without exposing secret values.
- Service deploy workflow — detects whether the selected Service declares Service Secrets, requires/decrypts the Service Sibling SOPS File only in that case, installs service-scoped Podman secrets, and passes only non-secret metadata to rendering.
- Inventory/fragment validation — rejects `_FILE` naming violations and environment conflicts between Service yaml, generated secret env vars, and Quadlet Fragment `Environment=` entries.
- Test fixtures and workflow tests — cover rendered output, validation failures, and missing Service Sibling SOPS File behavior.

**Acceptance criteria:**
- [x] Service Sibling SOPS File stores Service Secrets under top-level `secrets:`.
- [x] Container `secrets` entries reference `secrets.<name>` and declare an env var ending in `_FILE`.
- [x] Podman secret names are service-scoped, for example `fortress_<service>_<secret>`.
- [x] Rendered containers receive `_FILE` env vars pointing at `/run/secrets/<service-scoped-secret-name>`.
- [x] Rendered containers bind the service-scoped Podman secret without embedding the secret value.
- [x] Secret values never appear in rendered Quadlet units, plaintext Service yaml, command output, or plaintext env.
- [x] `service-deploy` requires the Service Sibling SOPS File only when containers declare Service Secrets.
- [x] Non-secret `env` is declared in Service yaml and rendered as container environment.
- [x] Validator rejects container secret env vars that do not end in `_FILE`.
- [x] Validator rejects conflicts where Service yaml `env`, generated secret `_FILE` env vars, or Quadlet Fragment `Environment=` entries override each other.
- [x] Golden-file and workflow tests cover secrets injection, missing Service Sibling SOPS File when secrets are declared, no Service Sibling SOPS File requirement when no secrets are declared, non-secret env rendering, and env conflict validation.

**Out of scope:**
- Native deploy substrate secret handling.
- Supporting applications that cannot consume secrets through the `_FILE` convention.
- Secret rotation policy or long-lived secret lifecycle commands beyond replacing the service-scoped Podman secret during deploy.
- Service destroy/delete behavior.
- Changing VM Sibling SOPS File SSH handling.
