## What to build

Install Service Secrets from the Service Sibling SOPS File as service-scoped Podman secrets and inject them into containers only through `_FILE` environment variables, while keeping non-secret env in the Service yaml.

## Acceptance criteria

- [ ] Service Sibling SOPS File stores Service Secrets under top-level `secrets:`
- [ ] Container `secrets` entries reference `secrets.<name>` and declare an env var ending in `_FILE`
- [ ] Podman secret names are service-scoped, e.g. `fortress_<service>_<secret>`
- [ ] Rendered containers receive `_FILE` env vars pointing at `/run/secrets/<service-scoped-secret-name>`
- [ ] Secret values never appear in rendered Quadlet units or plaintext env
- [ ] `service-deploy` requires the Service Sibling SOPS File only when containers declare Service Secrets
- [ ] Non-secret `env` is declared in Service yaml and rendered as container environment
- [ ] Validator rejects conflicts where Service yaml `env`, generated secret `_FILE` env vars, or Quadlet Fragment `Environment=` entries override each other
- [ ] Golden-file and workflow tests cover secrets injection, missing Service Sibling SOPS File when secrets are declared, and env conflict validation
