## What to build

Implement `just service-deploy service=<name>` as the operator workflow that deploys or redeploys one declared Quadlet Service onto its Backend VM, manages only fortress-owned runtime artifacts for that Service, and surfaces failures without requiring manual digging.

## Acceptance criteria

- [ ] `just service-deploy service=<name>` invokes a script/playbook that deploys only the selected declared Service
- [ ] Deploy requires the Backend VM Sibling SOPS File for SSH
- [ ] Deploy requires the Service Sibling SOPS File only when containers declare Service Secrets
- [ ] Deploying a Service in a Service Group ensures the shared group network exists but does not deploy other Services in the group
- [ ] Redeploy prunes obsolete fortress-rendered Quadlet units for the selected Service
- [ ] Redeploy prunes obsolete service-scoped Podman secrets using only the Service's fortress secret prefix
- [ ] Redeploy never prunes `/srv/services/<service>/` data; Service renames and Service Path changes require explicit operator data migration
- [ ] Redeploy performs a full Service restart in dependency order: stop containers in reverse topological order and start containers in topological order
- [ ] If a container fails to start, deploy aborts remaining starts, surfaces the failed unit name plus relevant status/journal guidance, and leaves rollback as an explicit operator action
- [ ] Workflow tests cover successful deploy, Service Group network handling, removed-container pruning, secret pruning boundary, data preservation, dependency-order restart, and start-failure reporting
