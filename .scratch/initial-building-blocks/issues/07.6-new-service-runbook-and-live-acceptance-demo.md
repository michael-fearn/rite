Document the Service authoring and deployment workflow and prove the Quadlet substrate with a live, real-world-shaped multi-container Service acceptance demo on a VM.

## Acceptance criteria

- [x] `runbooks/new-service.md` explains the Service yaml shape, Backend and hostname naming planes, Ingress defaults, Published Ports, Service Groups, Container Aliases, Service-owned volumes, Share-backed Volumes, Service Secrets, Quadlet Fragments, and Service Data Owner
- [x] Runbook explains that Share-backed Volumes reference a Mount Name already declared on the Backend VM; Service yaml does not declare NAS Endpoint, Dataset, Share, or protocol details directly
- [x] Runbook explains that `service-deploy` may validate Share-backed Volume subpaths but does not run NAS Reconcile, create NAS Shares, or create VM Mount units
- [x] Runbook explains that VM placement is the Service security boundary and issue-07 Quadlets are rootful system units

- [ ] Demo verifies the Service is reachable through its Ingress hostname when enabled and that same-network container communication works through Container Aliases
