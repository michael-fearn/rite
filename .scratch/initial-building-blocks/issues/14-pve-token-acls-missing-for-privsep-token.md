## Summary

Host Configure creates and stores the `tofu@pve!tofu` PVE API token, but the `proxmox_users` role only applies ACL roles to the user `tofu@pve`. The token is created with Proxmox privilege separation enabled (`privsep=1`), so the token has no effective permissions even though the user has the declared roles.

## Reproduction

On a configured Host such as `wintermute`, run:

```sh
ansible wintermute -i inventory/fortress.yaml -m ansible.builtin.command \
  -a 'pveum user token list tofu@pve'
```

Observed token output includes:

```text
tokenid  tofu
privsep  1
```

Then run:

```sh
ansible wintermute -i inventory/fortress.yaml -m ansible.builtin.command \
  -a 'pveum acl list --output-format json'
```

Observed ACL output grants roles to the user only:

```json
[
  {"path":"/","propagate":1,"roleid":"PVEVMAdmin","type":"user","ugid":"tofu@pve"},
  {"path":"/","propagate":1,"roleid":"PVEDatastoreAdmin, PVESDNAdmin","type":"user","ugid":"tofu@pve"}
]
```

Finally run:

```sh
ansible wintermute -i inventory/fortress.yaml -m ansible.builtin.command \
  -a 'pveum user token permissions tofu@pve tofu --output-format json'
```

Observed:

```json
{}
```

## Expected Behavior

The declared PVE API token principal should have effective permissions matching its declared token roles:

```yaml
proxmox:
  users:
    - name: tofu@pve
      roles: [PVEVMAdmin, PVEDatastoreAdmin, PVESDNAdmin]
      tokens:
        - id: tofu
          roles: [PVEVMAdmin, PVEDatastoreAdmin, PVESDNAdmin]
```

After Host Configure, `pveum user token permissions tofu@pve tofu --output-format json` should return non-empty privileges sufficient for OpenTofu VM lifecycle operations.

The parent PVE user's ACL roles should continue to be applied as they are today, but they are not the effective permission source for privilege-separated automation tokens. Scoped API tokens carry their own ACL roles so the operator can narrow token blast radius independently from the parent user.

## Code Pointers

- `ansible/roles/proxmox_users/tasks/main.yml` currently loops over `proxmox_users_declared | subelements('roles')` and runs `pveum aclmod / -user ... -role ...`.
- `scripts/host-configure` creates the token via `ansible/playbooks/host-token-create.yml` before running `ansible/playbooks/host-configure.yml`.
- `inventory/hosts/wintermute.yaml` declares token roles under `proxmox.users[].tokens[].roles`.

## Suggested Fix

Update `ansible/roles/proxmox_users/tasks/main.yml` so declared token roles are applied to the PVE API token principal as well as the user principal. Keep existing parent-user ACL behavior unchanged for this bug fix.

Use one command per declared token, with that token's roles comma-joined. The Proxmox command shape is:

```sh
pveum acl modify / --tokens tofu@pve!tofu --roles PVEVMAdmin,PVEDatastoreAdmin, PVESDNAdmin
```

The implementation should derive `tofu@pve!tofu` from `user.name` and `token.id`, and should use `token.roles` rather than assuming the token always inherits `user.roles`. Apply token ACLs at the root path only for this issue.

`inventory/hosts/_schema.json` already requires `roles` on every declared token, so the role can assume `token.roles` is present and non-empty for valid inventory.

For idempotency, follow the existing `proxmox_users` ACL style: declare the desired ACL on every run and report no Ansible change for already-converged ACL declaration (`changed_when: false`), rather than parsing `pveum acl list` before issuing the command.

## Design Decisions

- The **PVE API Token Principal** (`<user>@<realm>!<token>`) is the permission-bearing automation identity.
- Parent PVE user ACLs remain supported and should continue to be applied from `user.roles`.
- Token ACLs must come from `token.roles`; tokens do not inherit effective permissions from `user.roles` when privilege separation is enabled.
- Declared tokens without roles are invalid inventory, not intentionally permissionless tokens.
- Apply token ACLs once per token with comma-joined roles, not once per token-role pair.
- Token ACLs are root-only for this issue; per-token ACL paths are out of scope.
- Token ACL cleanup/pruning is out of scope when token roles change or tokens are removed.
- Versioned token naming for rotation (`tofu-vN`) is out of scope; derive the principal from the current `token.id` value.
