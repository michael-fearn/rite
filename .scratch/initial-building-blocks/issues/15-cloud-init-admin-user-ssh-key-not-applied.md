# Cloud-init does not apply VM admin user and SSH key for wintermute-demo

## Report

When running `just vm-up vm=wintermute-demo`, cloud-init does not appear to contain or apply the default VM admin user or the SSH public key needed for Ansible automation.

## Context

Fortress expects VM bring-up to follow Prepare -> OpenTofu apply -> Configure. Prepare writes the VM-specific public SSH key into the VM YAML. OpenTofu uploads per-VM cloud-init userdata. Cloud-init should create the default VM admin user from global inventory, authorize the prepared public key, set hostname/network intent, and leave the VM reachable so Configure can continue through Ansible.

This is a bug against the VM lifecycle contract, not a new feature. It touches the cloud-init first-boot boundary described by ADR-0010 and the OpenTofu/Ansible ownership boundary described by ADR-0002.

## What we established so far

- `inventory/vms/wintermute-demo.yaml` exists and includes `ssh_public_key`.
- `inventory/group_vars/all.yaml` sets `vm_admin_user: admin`.
- The OpenTofu VM partition module currently renders cloud-init userdata with `users`, `name: ${var.admin_user}`, `sudo`, and `ssh_authorized_keys`.
- `vm-configure` waits for SSH readiness and then `cloud-init status --wait`, so if the admin user or key is absent, Configure either cannot connect or is only succeeding through some unintended path.
- The completed wintermute demo issue recorded a successful `just vm-up vm=wintermute-demo` run on 2026-05-04, including a prior change from cloud-init user `fortress` to `admin`, so this may be a regression, stale Proxmox snippet/application state, or a false-positive configure path.
- Live diagnosis on 2026-05-06 confirmed the real `wintermute-demo` VM is running at VMID `8801` on Host `wintermute`, and Proxmox config includes `ipconfig0: gw=10.10.0.1,ip=10.10.0.201/24`.
- `just vm-shell wintermute-demo` derives the expected SSH target from Ansible Inventory, then fails at SSH with `No route to host` for `10.10.0.201`.
- The placement Host `wintermute` is reachable at `10.10.0.11`, so the failure is not a general operator-workstation route failure.
- The QEMU guest agent reports the VM NIC `enp6s18`, but no guest IP address.
- Inside the VM, `cloud-init status --long` reports `disabled-by-generator`, `/var/log/cloud-init.log` is absent, and `enp6s18` is `DOWN`.
- The guest sees no cloud-init CD-ROM device (`/dev/sr0`, `/dev/cdrom`, and `/dev/disk/by-label` are absent).
- QEMU reports the Proxmox cloud-init drive attached as `ide2`, but block stats show no reads from that device.

## Agent Brief

**Category:** bug
**Summary:** Make the VM first-boot cloud-init datasource visible to Debian q35/OVMF guests so VM networking, admin user, and SSH key are applied.

**Current behavior:**
Fortress provisions a VM with Proxmox cloud-init initialization data and a custom user-data snippet. For `wintermute-demo`, Proxmox reports the VM is running and has the declared static IP config, but the guest never sees a cloud-init datasource. Cloud-init disables itself during generator detection, the VM NIC stays down with no IP address, and SSH to the Ansible Inventory target fails.

**Desired behavior:**
New VMs provisioned from the fortress Template should see the cloud-init datasource on first boot. Cloud-init should run once, set the declared hostname, configure the VM admin user, authorize the prepared VM SSH public key, and apply the declared network address so the VM becomes reachable at its Ansible Inventory host.

**Key interfaces:**
- VM Inventory network declaration — continues to declare static address, gateway, bridge, model, and optional VLAN.
- Template hardware declaration — q35/OVMF and virtio-scsi Template settings must remain supported.
- OpenTofu VM provisioning contract — creates the VM, attaches the cloud-init datasource/user-data, and configures network initialization without moving ongoing configuration into cloud-init.
- VM lifecycle commands — `vm-up` should still perform Prepare -> OpenTofu apply -> Configure; `vm-shell` should work once the VM is prepared and reachable.

**Acceptance criteria:**
- [ ] A newly provisioned VM using the Debian q35/OVMF Template sees a cloud-init datasource at first boot.
- [ ] Cloud-init no longer reports `disabled-by-generator` on a freshly provisioned VM.
- [ ] The guest receives the static IP address declared in VM Inventory.
- [ ] The configured VM admin user exists after first boot.
- [ ] The VM admin user's authorized keys include the public key written by Prepare.
- [ ] `just vm-shell <vm>` reaches SSH for a freshly provisioned VM after cloud-init completes.
- [ ] `just vm-configure <vm>` connects using the VM Sibling SOPS File private key and does not depend on any template-baked or root fallback access.
- [ ] Tests or a runbook verification step cover the datasource visibility failure mode for q35/OVMF guests.

**Out of scope:**
- Changing the VM admin username policy.
- Moving ongoing Configure responsibilities from Ansible into cloud-init.
- Changing Host bootstrap SSH behavior.
- Keeping an already-misbooted VM disk and trying to repair cloud-init in place; reprovisioning is acceptable because cloud-init is first-boot only.
