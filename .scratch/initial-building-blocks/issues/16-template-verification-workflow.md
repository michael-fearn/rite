# Template verification workflow umbrella

## What to build

Build a gated real-VM Template verification workflow that provisions a temporary Template Verification VM from a selected Template on a selected Host, verifies the VM lifecycle contract from inside the VM with Ansible, and then destroys the VM through the normal VM lifecycle cleanup path.

This issue is now an umbrella/design record. The implementation work has been split into smaller agent-grabbable child issues.

The operator surface remains:

```sh
just template-verify host=wintermute template=debian-12-base
just template-verify host=all template=debian-12-base
just template-verify host=wintermute template=debian-12-base keep_on_fail=true
```

The ordinary automated test suite should use fake tools to verify the workflow wiring. The real Proxmox provisioning path is intentionally gated behind the operator command.

## Context

Fortress Templates live on specific Hosts. A Template is not verified globally; it is verified on each Host that declares it under `proxmox.templates`.

The workflow should dogfood the existing VM lifecycle rather than inventing a parallel path. It should generate a concrete temporary VM declaration, run the normal `vm-up` flow, run an Ansible verification playbook, and delegate cleanup to `vm-destroy delete_vm_yaml=true`.

This workflow formalizes a small VMID allocation policy:

- Ordinary VMs: `100`-`8899`
- Operational VMs: `8900`-`8999`
- Templates: `9000`-`9999`

The Template Verification VM is an Operational VM and should default to VMID `8901`.
