# Template Verification VM generation and preflights

## Parent

.scratch/initial-building-blocks/issues/16-template-verification-workflow.md

## Agent Brief

**Category:** enhancement
**Summary:** Generate concrete temporary Template Verification VM inventory and fail early when policy, Host, Template, or real Proxmox preflights are not satisfied.

**Current behavior:**
There is no workflow primitive that turns a selected Host and Template into a concrete Template Verification VM declaration. Template existence is only established during Template build or later VM provisioning, and there is no preflight that proves a selected Host has the Template declared and present in real Proxmox as `template: 1`.

**Desired behavior:**
Given a Host and Template, fortress should derive a complete temporary Template Verification VM declaration from inventory and the Template Verification Policy. The generated VM declaration should be concrete, use the reserved Template Verification VMID, mark itself as a generated Operational VM, disable backups, and include no placeholders.

Before writing generated inventory or provisioning anything, the workflow should fail clearly if the selected Host is missing, the selected Template is missing, the Host does not declare the Template under `proxmox.templates`, the policy lacks Host-specific storage or address data, bridge/gateway derivation is absent or ambiguous, or generated temporary artifacts already exist. It should also verify the real Proxmox Template on the Host by decrypting the Host SSH key and running `qm config <template_vmid>`, requiring `template: 1`.

**Key interfaces:**
- Template Verification Policy — supplies VMID, hardware sizing, disk size, per-Host storage, and per-Host address.
- Host inventory network data — `network.management_address` is matched to exactly one Host bridge `cidr`; that bridge supplies the verification bridge and gateway.
- Template inventory and Host Template declarations — prove that the selected Template exists and belongs to the selected Host before any provisioning.
- Real Proxmox Template preflight — checks `qm config <template_vmid>` on the selected Host and requires `template: 1`.
- Generated VM inventory — writes `inventory/vms/tmp-template-verify.yaml` and `inventory/vms/tmp-template-verify.sops.yaml` only after all preflights pass.

**Acceptance criteria:**
- [ ] The generation path refuses to start if `inventory/vms/tmp-template-verify.yaml` already exists.
- [ ] The generation path refuses to start if `inventory/vms/tmp-template-verify.sops.yaml` already exists.
- [ ] Missing Host, missing Template, and Template-not-declared-on-Host cases fail before generated files are written.
- [ ] Missing policy storage or address entries for the selected Host fail before generated files are written.
- [ ] Bridge and gateway derivation succeeds only when the Host management address matches exactly one Host bridge CIDR.
- [ ] Missing or ambiguous bridge/gateway derivation fails with a useful surfaced message.
- [ ] The real Proxmox Template preflight decrypts the Host SSH key and runs `qm config <template_vmid>` on the selected Host.
- [ ] The real Proxmox Template preflight fails clearly if the VMID is absent or is not marked `template: 1`.
- [ ] The generated VM declaration is concrete and includes VMID, placement, source Template, hardware, network, cloud-init hostname, backup disabled, SSH public key data needed by the lifecycle, and Operational VM lifecycle markers.
- [ ] The generated Sibling SOPS File has the expected sparse VM secret shape for the temporary VM.
- [ ] Fake-tool tests cover successful generation, existing temp artifact refusal, missing policy entries, ambiguous bridge derivation, missing Host/Template declarations, and real Template preflight failure.

**Out of scope:**
- Adding the final `just template-verify` operator command.
- Running the full `vm-up` path.
- Running the Ansible verification playbook.
- Destroying the temporary VM after verification.
