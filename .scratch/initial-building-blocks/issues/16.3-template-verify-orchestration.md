# Template verify orchestration

## Parent

.scratch/initial-building-blocks/issues/16-template-verification-workflow.md

## Agent Brief

**Category:** enhancement
**Summary:** Add the `template-verify` operator command that runs generation, normal VM lifecycle provisioning, verification, and cleanup with clear single-Host and all-Host reporting.

**Current behavior:**
Operators can build Templates and can run the normal VM lifecycle, but there is no single command that verifies a selected Template on a selected Host. There is also no `host=all` reporting mode for per-Host Template verification and no `keep_on_fail` behavior for preserving a failed Template Verification VM for inspection.

**Desired behavior:**
An operator can run `just template-verify host=<host|all> template=<template> keep_on_fail="false"`. For a single Host, the workflow should generate the temporary VM declaration, run the normal `vm-up tmp-template-verify` path, invoke the Template verification playbook, and then run `vm-destroy tmp-template-verify delete_vm_yaml=true` on success.

For `host=all`, the workflow should iterate Hosts sequentially and report each Host as `passed`, `failed`, or `skipped`. `skipped` is only valid when that Host does not declare the selected Template. Missing policy, failed preflight, failed provision, failed Configure, failed verification, or failed cleanup should report `failed` with useful messages.

On failure, cleanup should still run by default. With `keep_on_fail=true`, the temporary VM declaration, Sibling SOPS File, and real VM should be preserved for inspection.

**Key interfaces:**
- `template-verify` operator command — accepts `host`, `template`, and `keep_on_fail` arguments.
- Template Verification VM generation/preflight primitive — supplies the generated inventory and early failure behavior.
- VM lifecycle workflows — orchestration must use the normal `vm-up tmp-template-verify` and `vm-destroy tmp-template-verify delete_vm_yaml=true` paths.
- Template verification playbook interface — returns success or failure after checking the VM lifecycle contract.
- Summary reporting — gives each selected Host a `passed`, `failed`, or `skipped` result.

**Acceptance criteria:**
- [x] `just template-verify host=<host|all> template=<template> keep_on_fail="false"` exists and calls a backing workflow.
- [x] `host=<name>` verifies the selected Template only on the selected Host.
- [x] `host=all` iterates sequentially over every Host.
- [x] `host=all` reports each Host as `passed`, `failed`, or `skipped`.
- [x] A Host is marked `skipped` only when that Host does not declare the selected Template.
- [x] Missing policy, failed preflight, failed provision, failed Configure, failed verification, and failed cleanup are reported as `failed` with useful messages.
- [x] The workflow runs the full normal `vm-up tmp-template-verify` path before verification.
- [x] On `passed`, cleanup is performed by `vm-destroy tmp-template-verify delete_vm_yaml=true`.
- [x] On `failed`, cleanup still runs by default through `vm-destroy tmp-template-verify delete_vm_yaml=true`.
- [x] On `failed` with `keep_on_fail=true`, the temporary VM declaration, Sibling SOPS File, and real VM are preserved.
- [x] Cleanup failure is surfaced as a failure without hiding the original verification/provision failure context.
- [x] Fake-tool tests cover single-Host success, `host=all` passed/failed/skipped summaries, verification failure with default cleanup, verification failure with `keep_on_fail=true`, and cleanup failure reporting.

**Out of scope:**
- Adding the Template Verification Policy shape.
- Implementing the real Proxmox Template preflight.
- Defining the detailed Ansible assertions inside the verification playbook.
- Auto-building Templates as part of verification.
- Adding a Template-verification-specific auto-approve flag to the general `vm-up` operator command.
