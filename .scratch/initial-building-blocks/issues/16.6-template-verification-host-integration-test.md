# Template verification host integration test

## Parent

.scratch/initial-building-blocks/issues/16-template-verification-workflow.md

## Human Brief

**Category:** enhancement
**Summary:** Run the completed Template verification workflow against a real Host and record whether it satisfies the original workflow contract.

**Why this is ready for human:**
This step requires real Host and Proxmox access. It provisions a temporary Operational VM, connects to it with Ansible, and destroys it through the normal VM lifecycle cleanup path. The fast fake-tool suite has already covered workflow wiring; this issue is for the gated real-host acceptance pass.

**Current behavior:**
Implementation child issues 16.1 through 16.5 are complete. The repo exposes `just template-verify host=<host|all> template=<template> keep_on_fail="false"` and documents the operator flow, but the umbrella issue still needs a real Host integration result.

**Desired behavior:**
A human operator runs Template verification against a Host that declares the selected Template. The workflow should preflight the real Proxmox Template, generate the temporary Template Verification VM declaration, run the normal `vm-up tmp-template-verify` lifecycle, verify the VM lifecycle contract from inside the VM, and clean up with `vm-destroy tmp-template-verify delete_vm_yaml=true`.

**Suggested command:**

```sh
just template-verify host=wintermute template=debian-12-base
```

If failure inspection is needed:

```sh
just template-verify host=wintermute template=debian-12-base keep_on_fail=true
```

After inspecting a preserved failed VM:

```sh
just vm-destroy tmp-template-verify delete_vm_yaml=true
```

**Acceptance criteria:**

- [x] The selected Host declares the selected Template under `proxmox.templates`.
- [x] The selected Template exists on the Host and the preflight confirms it is marked `template: 1`.
- [x] `just template-verify host=<host> template=<template>` provisions `tmp-template-verify` through the normal VM lifecycle.
- [x] The Template verification playbook passes all in-VM lifecycle contract checks.
- [x] On success, the temporary VM and generated VM yaml/Sibling SOPS File are removed by cleanup.
- [x] The operator records the command output summary in this issue's comments.
- [x] Any failure is either fixed directly or filed as a separate bug with the failing command, surfaced error, and whether `keep_on_fail=true` preserved artifacts for inspection.

**Out of scope:**

- Changing the Template verification implementation unless the host run exposes a bug.
- Replacing fake-tool tests with real Proxmox tests in the default automated suite.
- Auto-building missing Templates as part of verification.
