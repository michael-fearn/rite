## Summary

The NFS live demo currently depends on an ordinary adopted Dataset named `test`, but an Acceptance Test demo should create, exercise, and clean up its own Ephemeral Dataset. When the `test` Dataset is missing, the live plan correctly refuses to recreate it, which blocks the remaining NFS integration demo acceptance criterion.

## Reproduction

With the `test` Dataset absent from TrueNAS and the demo VM still declaring a Mount against Dataset `test`, run:

```sh
./scripts/nas-reconcile-plan --live truenas
```

Observed behavior:

- The plan is blocked.
- `dataset_findings` includes `missing_dataset` for `test` at `/mnt/tank/test`.
- The desired NFS Share may be derived, but no write actions are produced because adopted Dataset findings block apply.

This is correct behavior for ordinary adopted Datasets, but it is the wrong setup for the NFS Acceptance Test demo.

## Agent Brief

**Category:** bug
**Summary:** Rewire the NFS live demo so it creates and cleans up an Ephemeral Dataset instead of depending on an operator-created adopted Dataset.

**Current behavior:**
The live NFS demo VM declares a Mount against Dataset `test`. That Dataset is ordinary fleet Inventory with adopted semantics, so NAS Reconcile validates that it already exists and refuses to create it when missing. This protects real data, but it means the demo cannot prove the full Acceptance Test path from Dataset creation through NFS Share reconciliation to VM systemd Mount verification.

The reconcile model already distinguishes adopted Datasets from Ephemeral Datasets, and Acceptance Test mode can permit `lifecycle: ephemeral` Datasets. The demo fixture and operator workflow have not been wired to use that acceptance contract.

**Desired behavior:**
The NFS Acceptance Test demo should use an explicitly acceptance-owned Ephemeral Dataset. The demo should be able to start with no preexisting Dataset on TrueNAS, create the Ephemeral Dataset in acceptance mode, create the derived fortress-owned NFS Share, configure the demo VM Mount, verify the VM sees a functional systemd-managed NFS Mount, and then clean up only resources that can be proven to be acceptance-owned.

Ordinary fleet Inventory must continue to reject Ephemeral Datasets unless an Acceptance Test path explicitly opts in. Ordinary adopted Datasets must remain protected: missing adopted Datasets should still block reconcile instead of being created.

**Key interfaces:**

- Dataset lifecycle semantics — adopted Datasets are protected and never created by routine reconcile; Ephemeral Datasets may be created and destroyed only in acceptance mode.
- NAS Reconcile acceptance mode — should include Ephemeral Datasets in desired Share derivation and produce Dataset create/delete actions only when the acceptance flag is present.
- Live TrueNAS adapter — should be able to create/delete Ephemeral Datasets and preserve the fortress ownership marker so cleanup can distinguish acceptance-owned resources from unmanaged resources.
- NFS demo VM Inventory — should reference the acceptance Dataset used by the demo Mount, not the ordinary adopted `test` Dataset.
- Live demo/runbook instructions — should show the acceptance-mode reconcile commands and the VM-side systemd Mount verification.

**Acceptance criteria:**

- [ ] The NFS live demo no longer depends on an operator-created adopted Dataset named `test`.
- [ ] The demo declares or otherwise supplies an Ephemeral Dataset dedicated to the Acceptance Test.
- [ ] Ordinary inventory validation continues to reject `lifecycle: ephemeral` unless the acceptance path explicitly opts in.
- [ ] Acceptance-mode live reconcile can create the Ephemeral Dataset when it is absent.
- [ ] Acceptance-mode live reconcile creates the derived fortress-owned NFS Share for the demo VM Mount.
- [ ] The demo VM can be configured so the declared Mount is rendered as a systemd `.mount` unit and becomes active.
- [ ] The demo verification checks systemd state and VM mount functionality through the public operator workflow.
- [ ] Cleanup removes only the fortress-marked Ephemeral Dataset and its fortress-owned NFS Share.
- [ ] Missing adopted Datasets still block ordinary reconcile and are not created.
- [ ] Automated tests cover the demo inventory contract and live adapter behavior for Ephemeral Dataset create/delete marker handling.

**Out of scope:**

- Creating, repairing, or deleting ordinary adopted Datasets.
- Deleting unmanaged TrueNAS Shares or Datasets.
- Broad live infrastructure CI beyond this NFS Acceptance Test path.
- Changing the general Dataset ownership convention for real NAS data.

## Notes

During the live demo, deleting the TrueNAS `test` Dataset exposed the mismatch: the code correctly treated `test` as adopted and refused to create it. That makes the demo safer, but not self-contained. The demo should move to the Ephemeral Dataset acceptance contract instead of asking the operator to manually recreate `test`.
