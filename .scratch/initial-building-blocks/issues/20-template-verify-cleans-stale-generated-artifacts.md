# Template verify cleans stale generated artifacts

## Report

`just template-verify host=straylight template=debian-13-base` failed before
verification because the generated Template Verification VM inventory file was
already present:

```text
/workspaces/fortress/inventory/vms/tmp-template-verify.yaml
```

The generation step refused to overwrite the file and exited with status 1.
The expected operator experience is that failed Template Verification runs do
not leave default cleanup-sensitive generated files behind in a way that blocks
the next run.

## Triage Notes

**What we've established so far:**

- This is a bug in the Template Verification workflow, not an enhancement request.
- The generated Template Verification VM is an Operational VM named `tmp-template-verify`.
- The generation primitive intentionally refuses to overwrite pre-existing generated VM inventory or a pre-existing generated VM Sibling SOPS File.
- That overwrite refusal is still valuable as a safety check, but it becomes a poor recovery path when the stale files were left behind by an earlier failed default run.
- The orchestration workflow currently runs cleanup after failures that happen after generation has succeeded, but a stale generated artifact causes the next run to fail during generation/preflight before ordinary cleanup has a chance to run.
- A prior real-host integration note recorded the same stale generated artifact symptom before manual cleanup.
- The current workspace has both generated Template Verification VM artifacts present, matching the reporter's failure mode.

**Reproduction signal:**

- Reporter observed `just template-verify host=straylight template=debian-13-base` failing before verification with exit code 1 because generated Template Verification VM inventory already existed.
- Static tracing shows the path: Template Verification orchestration invokes generation first; generation refuses if generated artifacts already exist; default cleanup only runs after generation succeeds.

## Agent Brief

**Category:** bug
**Summary:** Make Template Verification recover from stale generated Template Verification VM artifacts left by failed default runs.

**Current behavior:**
Template Verification creates deterministic generated artifacts for the temporary
Template Verification VM. If one of those artifacts is already present before a
new run starts, generation refuses to overwrite it and the whole workflow fails
during generation/preflight.

That refusal is safe for unknown or intentionally preserved files, but it means
a stale generated artifact left by a failed default run can block every later
Template Verification attempt until the Operator manually removes the file.

**Desired behavior:**
With the default failure policy, Template Verification should not leave generated
Template Verification VM artifacts in a state that blocks the next run.

If a previous failed run left generated artifacts behind and the Operator did
not ask to preserve them, a later Template Verification run should either clean
those stale generated artifacts before generation or otherwise provide an
operator-facing cleanup path as part of the workflow. The behavior must remain
conservative: only artifacts that are clearly generated for the Template
Verification VM may be removed automatically.

If `keep_on_fail=true` intentionally preserved a failed Template Verification
VM or its generated artifacts for inspection, the workflow should not silently
discard that state. It should keep preserving inspection semantics and report a
clear recovery instruction or require an explicit cleanup action before
overwriting/removing preserved artifacts.

The overwrite refusal should remain for ordinary VM inventory, non-generated
files, ambiguous files, and any state that cannot be proven to belong to the
generated Template Verification VM.

**Key interfaces:**

- Template Verification orchestration — should handle stale generated
  Template Verification VM artifacts before or during generation/preflight.
- Template Verification VM generation/preflight primitive — should preserve
  conservative overwrite safety while allowing safe cleanup of known generated
  Template Verification artifacts when appropriate.
- VM Destroy workflow — should still be used when a live Template Verification
  VM might exist and needs normal lifecycle cleanup.
- Generated artifact safety checks — should distinguish generated Template
  Verification VM artifacts from ordinary Inventory.
- Failure reporting — should clearly report the original failure and any cleanup
  failure without hiding either.
- `keep_on_fail` behavior — should continue to preserve generated artifacts and
  live resources when explicitly requested.

**Acceptance criteria:**

- [ ] A stale generated Template Verification VM inventory file from a default failed run does not cause the next default Template Verification run to fail solely because the file already exists.
- [ ] A stale generated Template Verification VM Sibling SOPS File from a default failed run does not cause the next default Template Verification run to fail solely because the file already exists.
- [ ] The workflow only removes or overwrites artifacts that are clearly generated for the Template Verification VM.
- [ ] Ordinary VM inventory files and ambiguous files are still protected by overwrite refusal.
- [ ] `keep_on_fail=true` continues to preserve the generated Template Verification VM artifacts and live resources for inspection.
- [ ] If preserved state cannot be safely cleaned automatically, the operator gets a clear actionable message instead of a generic overwrite refusal.
- [ ] Cleanup failure after an original Template Verification failure reports both the original failure and the cleanup failure.
- [ ] Automated tests cover stale generated inventory before generation, stale generated Sibling SOPS File before generation, conservative refusal for non-generated or ambiguous files, and `keep_on_fail=true` preservation semantics.

**Out of scope:**

- Deleting ordinary, non-generated VM inventory.
- Changing Template Verification Policy reservations.
- Changing the VM lifecycle Destroy contract for normal Operator-managed VMs.
- Adding cleanup support for unrelated Acceptance Test workflows.
