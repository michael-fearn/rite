# Selected VM tofu operations evaluate sibling VMs

## Report

`just template-verify host=straylight template=debian-13-base` failed while
cleaning stale Template Verification VM artifacts. The cleanup path invoked
`scripts/vm-destroy tmp-template-verify --delete-vm-yaml`, which then ran a
selected-VM OpenTofu destroy. OpenTofu failed before cleanup could complete:

```text
Error: Unsupported attribute

  on modules/vm-partition/main.tf line 76, in resource "proxmox_virtual_environment_file" "cloud_init_user_data":
  76:             - ${trimspace(each.value.ssh_public_key)}

This object does not have an attribute named "ssh_public_key".
```

The selected VM, `tmp-template-verify`, had been prepared and did have
`ssh_public_key`. The failing objects were other VMs on the same Host that have
not yet been prepared.

## Triage Notes

**What we've established so far:**

- This is a bug in selected-VM OpenTofu operation scoping, surfaced by Template Verification cleanup.
- `scripts/tofu-wrap` detects `-var selected_vm=<vm>` and generates provider/module coverage only for the selected VM's Host.
- `tofu/main.tf` still loads every VM YAML into `local.vms`.
- `fortress_tofu/generate.py` renders each Host VM partition as `for vm_name, vm in local.vms : vm_name => vm if vm.placement.host == "<host>"`.
- Therefore, a selected-VM plan/apply/destroy still passes all sibling VMs on that Host into `tofu/modules/vm-partition`.
- `tofu/modules/vm-partition/main.tf` renders cloud-init userdata with `each.value.ssh_public_key`, so any unprepared sibling VM on the Host can break an operation targeting a different VM.
- On `straylight`, the current unprepared sibling VMs missing `ssh_public_key` are `forgejo-vm`, `headscale-vm`, `identity-vm`, `internal-ingress-vm`, `observability-vm`, and `pbs-vm`.
- This makes `vm-up`, `vm-destroy`, and workflows built on them vulnerable to unrelated sibling VM inventory state.

**Reproduction signal:**

- Running `just template-verify host=straylight template=debian-13-base` failed with the unsupported `ssh_public_key` attribute error during stale artifact cleanup.
- Static tracing confirms the selected-VM wrapper narrows generated Hosts but not the `vms` map passed to the selected Host module.

## Agent Brief

**Category:** bug
**Summary:** Make selected-VM OpenTofu operations evaluate only the selected VM, not every sibling VM on the same Host.

**Current behavior:**
Selected-VM workflows pass `-var selected_vm=<vm>` to `scripts/tofu-wrap`. The
wrapper generates provider/module coverage only for the selected VM's Host, but
the generated Host module still receives every VM whose `placement.host` matches
that Host.

As a result, OpenTofu evaluates resources for sibling VMs even when the command
targets one VM. If any sibling VM is not ready for OpenTofu evaluation, such as a
declared-but-unprepared VM missing `ssh_public_key`, the selected operation can
fail before acting on the intended VM.

**Desired behavior:**
When `selected_vm` is set, generated VM partition HCL should pass only that VM
into the relevant Host module. Selected `vm-up`, `vm-destroy`, and cleanup paths
must not require unrelated sibling VMs on the same Host to be prepared.

Fleet-wide operations without `selected_vm` should keep evaluating all declared
VMs for each Host.

**Key interfaces:**

- `scripts/tofu-wrap` — detects `-var selected_vm=<vm>` and invokes `scripts/tofu-generate --selected-vm <vm>`.
- `fortress_tofu/generate.py` — renders `tofu/generated-vm-partitions.tf`.
- `tofu/main.tf` — defines `variable "selected_vm"` and `local.vms`.
- `scripts/vm-up` — runs selected-VM plan/apply targets for the VM resource and cloud-init file.
- `scripts/vm-destroy` — runs selected-VM targeted destroy and then removes the cloud-init file from state.
- `tofu/modules/vm-partition/main.tf` — currently assumes each VM passed to the module has `ssh_public_key`.
- Template Verification cleanup — depends on `vm-destroy tmp-template-verify --delete-vm-yaml` working even when sibling VM declarations are not prepared.

**Acceptance criteria:**

- [ ] With `selected_vm=tmp-template-verify`, the generated VM partition for the selected Host passes only `tmp-template-verify` into `module.vms_<host>.vms`.
- [ ] Selected-VM `tofu plan`, `tofu apply`, and `tofu destroy` do not evaluate sibling VMs on the same Host.
- [ ] A selected-VM operation succeeds past HCL evaluation when sibling VM YAML files on the same Host lack `ssh_public_key`.
- [ ] Fleet-wide generation without `selected_vm` continues to pass every declared VM to the appropriate Host partition.
- [ ] `vm-up <vm>` and `vm-destroy <vm>` continue to generate provider coverage only for the selected VM's Host.
- [ ] `template-verify` stale cleanup can call `vm-destroy tmp-template-verify --delete-vm-yaml` without being blocked by unrelated unprepared sibling VMs.
- [ ] Automated tests cover selected-VM generation filtering, fleet-wide generation behavior, and a selected-VM destroy or plan fixture with an unprepared sibling VM on the same Host.

**Out of scope:**

- Preparing all declared VMs on a Host as a prerequisite for selected operations.
- Making `ssh_public_key` optional for normal VM provisioning.
- Changing the VM lifecycle contract that `vm-prepare` writes plaintext public keys to VM YAML and encrypted private keys to the Sibling SOPS File.
- Changing Template Verification assertions or Template Verification Policy.
