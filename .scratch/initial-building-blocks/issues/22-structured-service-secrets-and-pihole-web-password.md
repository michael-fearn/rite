## What to build

Adopt the structured Service Secret shape for service-level secrets and use it to set the Pi-hole web/API password for `dns-primary`.

## Agent Brief

**Category:** enhancement
**Summary:** Enforce structured Service Secrets and wire the Pi-hole web/API password through `WEBPASSWORD_FILE`.

**Current behavior:**
Service yaml can declare Service Secrets with `secrets.<purpose>` references and app-native `_FILE` environment variables. The deploy workflow currently treats the matching SOPS entry as the secret value itself, so a scalar `secrets.<purpose>` value works. The Pi-hole Service exposes its admin HTTP port for operator use, but it does not yet declare a Service Secret for the Pi-hole web/API password.

The resolved domain decision is that a Service Secret is a structured entry in the Service's single Sibling SOPS File. Each entry stores secret bytes under `value` and keeps `created` and `version` metadata alongside it. The Service yaml uses a Fortress purpose name such as `web_api_password`, while the container receives the Service application's native `_FILE` variable name.

**Desired behavior:**
Service Secret deployment should require the structured shape:

```yaml
secrets:
  web_api_password:
    created: 2026-05-12T00:00:00Z
    version: 1
    value: "<secret bytes>"
```

When a Service declares a secret reference, the deploy workflow should preflight the decrypted Service SOPS shape before remote execution. Missing entries, scalar legacy entries, missing `created`, missing `version`, or missing `value` should fail locally with a clear Fortress error that does not print secret values.

Only the nested `value` should be installed as the service-scoped Podman secret. Metadata must not be installed as secret bytes and must not appear in rendered Quadlet units.

For `dns-primary`, Pi-hole should receive the `web_api_password` Service Secret through Pi-hole's app-native `WEBPASSWORD_FILE` environment variable. The actual password value should be generated, stored only encrypted in the Service Sibling SOPS File, and not printed in issue notes, tests, command output, or docs.

**Key interfaces:**
- Service Secret SOPS shape — top-level `secrets:` map, purpose-named entries, and required `created`, `version`, and `value` fields.
- Service deploy workflow — detects declared Service Secret references, validates the decrypted SOPS shape locally, and extracts only the nested `value` for Podman secret installation.
- Quadlet rendering — continues to render service-scoped Podman secret bindings and app-native `_FILE` environment variables without exposing secret bytes.
- Pi-hole Service declaration — references `secrets.web_api_password` and maps it to `WEBPASSWORD_FILE`.
- Service Secret tests and acceptance fixtures — use the structured shape rather than scalar legacy values.

**Acceptance criteria:**
- [ ] Service Secret installation extracts `secrets.<purpose>.value`, not `secrets.<purpose>`.
- [ ] `service-deploy` preflights declared Service Secrets against the decrypted Service SOPS File before invoking the remote Ansible playbook.
- [ ] Preflight errors clearly identify the missing or malformed Service Secret entry without printing secret values.
- [ ] Scalar legacy Service Secret entries are rejected rather than supported as a compatibility path.
- [ ] The acceptance/demo Service Secret fixture uses `created`, `version`, and `value`.
- [ ] `dns-primary` declares a `web_api_password` Service Secret and maps it to Pi-hole's `WEBPASSWORD_FILE`.
- [ ] The `dns-primary` Service Sibling SOPS File exists, is encrypted, and contains `secrets.web_api_password.created`, `secrets.web_api_password.version`, and `secrets.web_api_password.value`.
- [ ] Rendered Quadlet output for Pi-hole contains the `WEBPASSWORD_FILE` path but not the password value.
- [ ] Relevant unit/workflow tests pass.
- [ ] The DNS runbook or service-secret docs mention how the Pi-hole web/API password is represented without disclosing the password.

**Out of scope:**
- Adding per-secret SOPS files.
- Supporting scalar legacy Service Secret entries.
- Building a general service-secret rotation command.
- Changing the Pi-hole admin HTTP exposure model or routing it through Ingress.
- Running the live Pi-hole Web UI login test; live DNS/Web UI proof remains operator-controlled.
