## What to build

Add the importable **Operator Workflow Runner** foundation around an inspectable **Operator Workflow Plan**.

The runner should execute ordered command phases and explicit confirmation gates, return structured workflow results, and standardize failure details without taking ownership of any workflow-specific ceremony rules. Existing workflow scripts may keep their current implementation in this slice, but the new module must be usable by later VM Lifecycle Convergence, Service Launch, and Host Readiness migrations.

## Acceptance criteria

- [ ] A new importable module exposes an **Operator Workflow Plan** containing ordered command phases and distinct confirmation gates.
- [ ] A command phase can declare a stable id, display name, command, diagnostic label, streaming preference, and failure policy.
- [ ] A confirmation gate is a distinct plan step that requires exact operator input unless auto-confirm is enabled.
- [ ] The runner executes plan steps in order and stops immediately on failed phases whose failure policy is stop.
- [ ] The runner continues after failed phases whose failure policy is continue, records the failure, and returns a non-success workflow result.
- [ ] Streaming command phases prefix console output by phase and retain a bounded output tail for diagnostics.
- [ ] Non-streaming command phases capture stdout and stderr for diagnostics.
- [ ] Each phase result includes status, command, return code, captured stdout/stderr where available, captured tail where available, and standardized failure detail.
- [ ] Failure detail prefers stderr then stdout then `exit N` for non-streaming phases.
- [ ] Failure detail prefers captured tail then stderr then `exit N` for streaming phases.
- [ ] Fast unit tests cover successful execution, failed stop policy, failed continue policy, confirmation denial, auto-confirm, streaming prefix output, tail capture, and failure detail precedence.
