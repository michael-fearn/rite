## Agent Brief

**Category:** enhancement
**Summary:** Extend the Service-layer Acceptance Test to prove that a TrueNAS NFS service reload preserves existing VM Mounts and Service Share-backed Volume consumption.

**Current behavior:**
The Service-layer Acceptance Test provisions disposable Operational VMs, reconciles an Ephemeral Dataset and Derived NFS Share, deploys generated Services, and verifies baseline Mount, Service, Published Port, Container Alias, Native Service, and Service Secret behavior. NAS Reconcile can reload the TrueNAS NFS service when it changes NFS Shares, but the acceptance workflow does not explicitly prove that already-established Mounts and Share-backed Volumes remain usable after that reload.

**Desired behavior:**
After the existing baseline Service-layer verification passes, the workflow should explicitly reload the TrueNAS NFS service through the Acceptance NAS Credential. It should then prove fresh data flow through already-established Mounts and the already-running Service without remounting VMs, redeploying Services, or orchestrating container restarts.

The post-reload proof should write fresh content through the Primary Acceptance VM's existing Mount, read it from the Peer Acceptance VM's existing Mount, update the Service-served file through the Mount, and verify from the Peer Acceptance VM that the already-running Service serves the fresh content through its Share-backed Volume.

If the explicit reload call fails, the workflow should behave like a verification failure: report the reload error and run default cleanup unless `keep_on_fail=true`.

**Key interfaces:**
- Service-layer Acceptance Test workflow — should add an explicit post-baseline NFS reload phase and a post-reload verification phase.
- Acceptance NAS Credential — should be used for the explicit reload operation because this acceptance workflow already owns Ephemeral Dataset and NFS acceptance mutations.
- TrueNAS API client capability — should expose or reuse the operation that calls TrueNAS `service.reload` for the `nfs` service.
- SSH verification checks — should prove fresh post-reload file create/read/delete behavior through existing VM Mounts.
- Peer-to-Primary HTTP verification — should prove the already-running Service serves fresh post-reload content from the Share-backed Volume.
- Cleanup behavior — should preserve the current default cleanup behavior and `keep_on_fail=true` inspection path.

**Acceptance criteria:**
- [ ] Baseline Service-layer verification runs before the NFS reload phase.
- [ ] The workflow explicitly invokes a TrueNAS NFS service reload using the Acceptance NAS Credential.
- [ ] The reload phase does not depend on NAS Reconcile producing a Share mutation.
- [ ] Post-reload verification writes a fresh marker through the Primary Acceptance VM's existing Mount.
- [ ] Post-reload verification reads the fresh marker through the Peer Acceptance VM's existing Mount.
- [ ] Post-reload verification updates the Service-served `index.html` through the Primary Acceptance VM's existing Mount.
- [ ] Post-reload verification curls the already-running Service from the Peer Acceptance VM and observes the fresh post-reload content served through the Share-backed Volume.
- [ ] Fresh post-reload marker files are removed during verification or cleanup.
- [ ] The test does not assert stable container IDs, PIDs, or systemd invocation IDs across the reload.
- [ ] Reload failure reports a clear reload-phase error and still runs generated-resource cleanup by default.
- [ ] `keep_on_fail=true` preserves generated inventory and live resources after reload or post-reload verification failure.
- [ ] Automated tests cover the new phase ordering, explicit reload invocation, expected post-reload SSH/curl commands, reload failure handling, and `keep_on_fail=true` behavior.

**Out of scope:**
- Hard restart or stop/start testing of the TrueNAS NFS service.
- Proving stable container process identity across reload.
- Replacing NAS Reconcile or changing ordinary Share reconciliation semantics.
- Changing Dataset ACL, mode, mapped UID/GID, or non-root filesystem semantics.
- Adding an ADR unless implementation reveals a new hard-to-reverse architectural decision.
