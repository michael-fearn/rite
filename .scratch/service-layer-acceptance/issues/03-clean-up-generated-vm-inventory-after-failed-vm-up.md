# Clean up generated VM inventory after failed Service-layer vm-up

## Report

Failed Service-layer Acceptance Test runs can leave `tmp-service-primary.yaml` and `tmp-service-primary.sops.yaml` behind even when `keep_on_fail=false`.

The observed failure is specifically around the Primary Acceptance VM generated artifacts. After the failed run, the next Service-layer Acceptance Test refuses to start because the generated Primary Acceptance VM YAML and VM Sibling SOPS File already exist.

## Triage Notes

**What we've established so far:**

- The Service-layer Acceptance Test defaults `keep_on_fail=false`, so failed runs should attempt to remove generated Acceptance resources.
- The workflow tracks generated VM names in a `provisioned` collection before each `vm-up` invocation has actually succeeded.
- Cleanup treats any VM in that collection as live/provisioned and routes it through the VM Destroy workflow with generated YAML deletion enabled.
- If `vm-up` fails during Prepare or another early phase after creating the generated VM Sibling SOPS File but before a live VM exists, the VM Destroy path may fail before reaching inventory-file deletion.
- In that case, generated VM inventory artifacts can remain in the repo and block later runs through the existing overwrite preflight.
- This is not covered by the completed generated Acceptance artifact cleanup command issue, because that command is recovery tooling. The Acceptance Test workflow itself should still clean up after default failed runs where possible.

**Reproduction signal:**

- Static code tracing shows the bug path: failed `vm-up` for `tmp-service-primary` after generated inventory exists causes failure cleanup to classify the Primary Acceptance VM as provisioned even though `vm-up` did not complete.
- Existing tests cover verification/reload failures after successful VM provisioning, but do not cover `vm-up` failure after VM Prepare creates a Sibling SOPS File and before a destroyable VM exists.

## Agent Brief

**Category:** bug
**Summary:** Ensure Service-layer Acceptance Test failure cleanup removes generated Primary and Peer Acceptance VM inventory artifacts when VM provisioning fails before a live VM exists.

**Current behavior:**
When the Service-layer Acceptance Test fails during a generated VM provisioning phase, the workflow may preserve generated VM YAML and VM Sibling SOPS Files even though `keep_on_fail=false`.

The likely failure mode is that the workflow marks an Acceptance VM as provisioned before the `vm-up` command for that VM has completed successfully. Default failure cleanup then sends that VM through the VM Destroy workflow. If `vm-up` failed before a live VM resource exists, VM Destroy can fail before deleting the generated VM YAML and VM Sibling SOPS File. Those leftover generated artifacts cause the next Service-layer Acceptance Test run to refuse to start because it will not overwrite existing generated Acceptance artifacts.

**Desired behavior:**
With `keep_on_fail=false`, the Service-layer Acceptance Test should clean up generated VM inventory artifacts after any failed provisioning attempt.

If `vm-up` fails before a generated Acceptance VM is actually provisioned, cleanup should remove that VM's generated YAML and VM Sibling SOPS File directly, using the same generated-artifact safety checks as other Acceptance cleanup paths where appropriate.

If `vm-up` succeeds for a generated Acceptance VM and a later phase fails, cleanup should continue to destroy the live VM through the VM Destroy workflow and delete the generated VM YAML and VM Sibling SOPS File afterward.

If cleanup itself fails, the workflow should continue to report both the original failure and the cleanup failure.

`keep_on_fail=true` should continue to preserve generated inventory and live resources for inspection.

**Key interfaces:**

- Service-layer Acceptance Test provisioning state — should distinguish "attempted VM provisioning" from "VM provisioning succeeded and may need live VM Destroy".
- VM Destroy workflow — should still be used for generated Acceptance VMs that were successfully provisioned.
- Generated Acceptance artifact cleanup behavior — should remove generated VM YAML and VM Sibling SOPS Files directly when no live VM Destroy is appropriate.
- Failure reporting — should preserve the existing behavior of reporting the original failed phase and any cleanup failure.

**Acceptance criteria:**

- [ ] If Primary Acceptance VM provisioning fails before `vm-up` succeeds and `keep_on_fail=false`, default failure cleanup removes the generated Primary Acceptance VM YAML.
- [ ] If Primary Acceptance VM provisioning fails before `vm-up` succeeds and `keep_on_fail=false`, default failure cleanup removes the generated Primary Acceptance VM Sibling SOPS File.
- [ ] The same cleanup behavior applies to Peer Acceptance VM provisioning failures.
- [ ] A generated Acceptance VM is only routed through VM Destroy after its `vm-up` command has completed successfully.
- [ ] After a later phase fails following successful VM provisioning, cleanup still invokes VM Destroy for successfully provisioned generated Acceptance VMs.
- [ ] `keep_on_fail=true` still preserves generated VM YAML, VM Sibling SOPS Files, generated Service artifacts, and live resources for inspection.
- [ ] Cleanup failure after an original provisioning failure reports both the original failed phase and the cleanup failure.
- [ ] Automated tests cover a `vm-up` failure that occurs after generated VM YAML and VM Sibling SOPS File creation but before successful provisioning, and assert those generated files are removed by default.

**Out of scope:**

- Changing the manual generated Acceptance artifact cleanup command.
- Deleting ordinary, non-generated VM inventory.
- Changing VM Destroy behavior for normal Operator-managed VMs.
- Changing `keep_on_fail=true` preservation semantics.
