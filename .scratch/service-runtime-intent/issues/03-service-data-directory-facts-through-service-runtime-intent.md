# Service Data Directory facts through Service Runtime Intent

Status: ready-for-agent
Category: enhancement
Date: 2026-05-19

## Parent

`.scratch/service-runtime-intent/PRD.md`

## What to build

Move Service-owned volume and Service Data Directory runtime meaning into Service Runtime Intent. Fleet-wide analysis should expose facts for Service-owned bind mounts under `/srv/services/<service>/`, including the declared Service Path, resolved VM-local path, container path, effective access mode, and Service Data Owner when declared.

Service Deploy and Quadlet rendering should consume these facts where they currently discover Service Data Directories from raw Service yaml. Preserve current behavior: Service Data Directory contents remain unpruned, Service Data Owner applies only to Service-owned paths, and generated Service Data Files remain Adapter-owned.

## Acceptance criteria

- [ ] Service Runtime Intent exposes Service-owned volume facts identifying Service, container, volume index, Service Path, resolved VM-local path, container path, effective access mode, and Service Data Owner when declared.
- [ ] Service Runtime Intent exposes de-duplicated Service Data Directory facts derived from Service-owned volume facts.
- [ ] Service Data Owner is applied only to Service Data Directory facts and not to Share-backed Volume facts.
- [ ] Service Deploy still creates the same `fortress_service_data_directories` entries for Service-owned volumes.
- [ ] Service Deploy does not prune Service Data Directory contents as part of this migration.
- [ ] Quadlet rendering still produces unchanged `Volume=` lines for Service-owned volumes.
- [ ] Observability generated Service Data Files and Application Configuration Artifacts remain Adapter-owned and are not moved into Service Runtime Intent.
- [ ] Direct Service Runtime Intent tests cover resolved Service Data Directory paths, duplicate directory de-duplication, effective access modes, and Service Data Owner application.
