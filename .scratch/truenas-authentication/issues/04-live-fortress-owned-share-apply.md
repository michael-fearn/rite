# Live Fortress-owned Share apply

## What to build

Wire live `--apply` to execute the existing NAS Reconcile write actions against TrueNAS NFS Shares for the selected NAS Endpoint. The workflow may create, update, and delete Fortress-owned Shares, but must not mutate ordinary Datasets.

## Acceptance criteria

- [x] `scripts/nas-reconcile-plan --live truenas --apply` executes planned `create_nfs_share`, `update_nfs_share`, and `delete_nfs_share` actions through the TrueNAS client.
- [x] Apply refuses to create, update, delete, or repair ordinary Adopted Datasets.
- [x] Ordinary live apply refuses any Dataset write action (`create_dataset`, `update_dataset`, or `delete_dataset`) before contacting TrueNAS unless the workflow is in explicit acceptance-ephemeral mode.
- [x] Share writes include the durable Fortress ownership marker used by existing planning logic.
- [x] If a TrueNAS write fails, the workflow stops at the first failed operation, returns non-zero, and reports the operation class and Share target.
- [x] Failed apply does not attempt rollback or compensating delete/update; the next live plan shows remaining drift.
- [x] Tests prove forward-retry behavior with a fake client that fails mid-apply.

## Agent Brief

**Category:** enhancement
**Summary:** Execute live NAS Reconcile apply actions against TrueNAS for Fortress-owned NFS Shares.

**Current behavior:**
NAS Reconcile can plan from live TrueNAS reality for a selected NAS Endpoint, and fixture-backed apply can compute and record write actions for missing, drifted, and stale Fortress-owned NFS Shares. Live apply still does not execute those Share writes through the TrueNAS client; it uses the same recording behavior as offline apply.

The existing safety model already distinguishes ordinary Adopted Datasets from acceptance-only Ephemeral Datasets. Ordinary NAS Reconcile must refuse Dataset write actions before contacting TrueNAS, and live apply should only mutate Fortress-owned Shares.

**Desired behavior:**
When the operator runs live apply for a NAS Endpoint, NAS Reconcile should load live reality, compute the same plan shape as fixture-backed mode, and execute only planned Fortress-owned NFS Share writes through the TrueNAS client. Supported live Share operations are create, update, and delete.

Created and updated Shares must carry the durable Fortress ownership marker used by the planner so later live plans recognize them as Fortress-owned. Deleting should be limited to Shares recognized as Fortress-owned by that same ownership contract.

If a TrueNAS write fails, apply should stop immediately, return non-zero, and report the operation class and Share target without attempting rollback or compensating operations. The next live plan should remain the retry mechanism by showing whatever drift remains.

Ordinary live apply must never create, update, delete, or repair Adopted Datasets. Any Dataset write action in ordinary live apply should be refused before contacting TrueNAS. Acceptance-only Ephemeral Dataset mutation remains separate and requires the explicit acceptance workflow.

**Key interfaces:**
- `scripts/nas-reconcile-plan` command behavior — live `--apply` should use the selected NAS Endpoint's live client for Share writes while preserving fixture-backed apply behavior.
- TrueNAS client abstraction — should expose live NFS Share create, update, and delete operations matching the planner's write action semantics.
- NAS Reconcile write-action execution — should stop on the first write failure, surface a useful operator error, and leave rollback actions empty.
- Fortress-owned Share payload shape — should include the durable ownership marker consumed by existing planning logic.
- Dataset safety guard — ordinary live apply should reject Dataset write actions before network access unless the explicit acceptance-ephemeral mode is active.
