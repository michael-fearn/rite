# Live NAS Reconcile rejects SOPS-backed NAS Endpoint without api_token_env

## Summary

`scripts/nas-reconcile-plan --live truenas` currently refuses the real `inventory/nas/truenas.yaml` because it no longer declares `api_token_env`, even though the NAS Reconcile Credential has moved to `inventory/nas/truenas.sops.yaml`.

## Reproduction

From the repo root:

```bash
./scripts/nas-reconcile-plan --live truenas
```

Observed:

```text
NAS Endpoint truenas must declare api_token_env
```

> *This was generated by AI during triage.*

## Agent Brief

**Category:** bug
**State:** ready-for-agent
**Summary:** Resolve the drift between live NAS Reconcile's plaintext `api_token_env` requirement and the current SOPS-backed NAS Endpoint credential model.

**Current behavior:**
`scripts/nas-reconcile-plan --live truenas` exits before decrypting the endpoint's Sibling SOPS File because `_load_live_reality` requires `inventory.nas_endpoints["truenas"]["api_token_env"]`.

The real endpoint file currently contains only:

```yaml
name: truenas
management_address: 10.10.0.15
share_address: 10.40.0.15
```

The credential material now lives in `inventory/nas/truenas.sops.yaml` under `api_credentials.reconcile.value` and `api_credentials.acceptance.value`.

There is also a regression test that encodes the old expectation: `tests/test_nas_reconcile_plan.py::test_live_plan_requires_api_token_env_before_decrypting_credential`.

**Desired behavior:**
Live NAS Reconcile should follow the current SOPS-backed NAS Endpoint credential model and should not fail simply because the plaintext NAS YAML lacks `api_token_env`.

The implementation should make one coherent choice and update code, schema, fixtures, docs, and tests together:

- If `api_token_env` is still part of the intended non-secret NAS Endpoint interface, restore it consistently in the real inventory and keep it documented as plaintext configuration.
- If the move to SOPS means `api_token_env` should no longer be required in `inventory/nas/<endpoint>.yaml`, remove the plaintext requirement from the live command path and derive a private child-process environment variable internally.

Given the current real inventory and the user's report, the likely fix is the second option.

**Codebase context:**
- `scripts/nas-reconcile-plan` decrypts `api_credentials.reconcile.value` from `inventory/nas/<endpoint>.sops.yaml`, then exports it to a child process environment variable before invoking `fortress_nas.truenas_reality`.
- `inventory/nas/_schema.json` still requires `api_token_env`.
- `docs/architecture.md` still shows `api_token_env: TRUENAS_API_TOKEN` in the NAS Endpoint declaration example.
- `docs/adr/0019-truenas-api-authentication-uses-operator-environment.md` says the ordinary credential path is `api_credentials.reconcile.value` in the endpoint Sibling SOPS File and that the workflow exports the decrypted value into an environment variable named by `api_token_env`.
- The live command currently reports `connection.<endpoint>.credential_source` as `inventory/nas/<endpoint>.sops.yaml:api_credentials.reconcile.value`, which should remain credential-safe.

**Acceptance criteria:**
- [ ] `scripts/nas-reconcile-plan --live truenas` no longer fails with `NAS Endpoint truenas must declare api_token_env` against the current real inventory.
- [ ] Live NAS Reconcile still decrypts only `api_credentials.reconcile.value` from the endpoint Sibling SOPS File for ordinary live plan/apply.
- [ ] The decrypted NAS Reconcile Credential is not printed in stdout, stderr, logs, plan output, or temporary files.
- [ ] Missing endpoint Inventory, missing endpoint Sibling SOPS File, missing SOPS credential material, failed SOPS extraction, and live TrueNAS connection/preflight failures still produce clear operator errors.
- [ ] `inventory/nas/_schema.json`, fixture NAS endpoint YAML, `docs/architecture.md`, ADR 0019, and tests agree on whether `api_token_env` is required plaintext configuration or an internal implementation detail.
- [ ] Replace or update `test_live_plan_requires_api_token_env_before_decrypting_credential` so the test suite protects the chosen credential model instead of the stale one.

**Suggested implementation notes:**
- Prefer a small helper in `scripts/nas-reconcile-plan` for the runtime environment variable name if removing plaintext `api_token_env`; for example derive a stable uppercase name from the endpoint such as `FORTRESS_NAS_RECONCILE_TRUENAS_TOKEN`.
- Keep the child process boundary intact: `fortress_nas.truenas_reality` can continue to receive an environment variable name rather than a credential value on argv.
- Update expected connection metadata carefully. If `api_token_env` disappears from the public Inventory contract, plan output probably should omit it too and continue to expose only `credentials: operator_environment` plus `credential_source`.

**Out of scope:**
- Changing the encrypted credential paths under `api_credentials.reconcile.value` or `api_credentials.acceptance.value`.
- Introducing plaintext API token values in NAS Endpoint Inventory.
- Changing the TrueNAS API client behavior or the NAS Reconcile planning semantics.
- Creating, updating, or deleting ordinary Datasets.
